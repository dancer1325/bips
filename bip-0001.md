# What is a BIP (Bitcoin Improvement Proposal) ?

* BIP
  * == design document /
    * describe (technical + rationale) a NEW feature about
      * Bitcoin OR
      * 's processes or environment
    * provide
      * information -- to the -- Bitcoin community
  * 's goal
    * mechanisms / 
      * propose new features
      * collect issue's community input
      * document the design decisions
  * 's author
    * responsible for
      * building consensus | community
      * documenting dissenting opinions
  * == text files / hosted | versioned repository

# BIP Types -- `Type:` --

* Standards Track
  * change / affects | MOST OR ALL Bitcoin implementations
    * _Examples:_ 
      * change the network protocol
      * change block or transaction validity rules
      * change or add the interoperability of applications / use Bitcoin
  * == design document + reference implementation
    * reference implementation
      * recommendations
        * AFTER being reviewed and accepted, create it
      * == code OR patch OR URL to that

* Informational
  * == Bitcoin design issue OR general guidelines OR information to the Bitcoin community
    * ‚ùå!= NEW feature‚ùå
    * ‚ùå!= Bitcoin community consensus OR recommendation‚ùå
      * -> ‚ö†Ô∏èusers and implementers can ignore them‚ö†Ô∏è

* Process
  * around Bitcoin
    * ‚ùå!= Bitcoin codebase‚ùå
    * informative OR propose a change
  * vs Standards Track BIPs
    * ‚ùå!= topics‚ùå
  * ‚ö†Ô∏èrequirements‚ö†Ô∏è
    * community consensus
      * ‚ùå!= recommendations‚ùå
      * ‚ùå!= Informational‚ùå
  * _Examples:_
    * | decision-making process,
      * add procedures,
      * add guidelines,
      * add changes
    * | Bitcoin development,
      * adjust tools or environment 
    * meta-BIP
      * == about BIP process

# What are BIP's parts?

* [Preamble](#bip-header-preamble)
  * _Example:_ top of this file

* Abstract
  * technical issue / is addressed
    * ~200 word
  * _Example:_ [BIP0002's Abstract](bip-0002.md#abstract)

* Copyright/public domain
  * explicitly label
  * place | public domain ( == version | this BIP)
  * license | Open Publication License
  * _Example:_ [BIP0002's Copyright](bip-0002.md#copyright)

* Specification
  * if it's a NEW feature -> describe the syntax + semantics 
  * detailed enough / can be implemented -- by -- any current Bitcoin platform (Satoshi, BitcoinJ, bitcoin-js, libbitcoin)
  * _Example:_ [BIP0008's Specification](bip-0008.md#specification)

* Motivation
  * == why the existing protocol specification is inadequate -- to -- address the problem
    * if insufficient motivation -> may be rejected
  * BIPs' critical part / want to change the Bitcoin protocol
  * _Example:_ [BIP0008's Motivation](bip-0008.md#motivation)

* Rationale 
  * == what motivated the design + particular design decisions + alternative designs ALREADY discussed
    * == provide evidence of consensus
  * _Example:_ [BIP0003's Rationale](bip-0003.md#rationale)

* Backwards Compatibility 
  * use cases
    * BIPs / introduce backwards incompatibilities
  * == backwards incompatibilities + their severity + how to deal with them
    * if insufficient backwards compatibility treatise -> may be rejected
  * _Example:_ [BIP0008's Backwards compatibility](bip-0008.md#backwards-compatibility)

* Reference Implementation 
  * ‚ö†Ô∏èrequired BEFORE getting "Final" status‚ö†Ô∏è 
  * ‚ùåNOT required before getting "Accept" status‚ùå
  * recommendations
    * write AFTER
      * finish the specification and rationale
      * reach consensus
  * != final implementation (+ test code + documentation)
  * _Example:_ [BIP0008's Reference implementation](bip-0008.md#reference-implementation-)

# BIP Editor's responsibilities & workflow

* responsible for
  * checking BIP
    * was PREVIOUSLY sent -- to the -- Bitcoin development mailing list
    * is ready (sound and complete)
      * if it's 
        * NOT ready -> send it back -- to the -- author
        * ready -> create a PR | [this repo](https://github.com/bitcoin/bips)
    * 's ideas make technical sense
      * ‚ùåALTHOUGH they do NOT seem to be accepted‚ùå
    * 's title describe the content
  * editing BIP's
    * language (spelling, grammar, sentence structure, etc.)
    * markup (for reST BIPs)
    * code style (_Example:_ [BIP 8](bip-0008.md))
  * assign BIP numbers
    * NORMALLY, 
      * NEXT AVAILABLE one
    * SOMETIMES,
      * special/joke number
  * change BIP's status
  * reject BIP proposals
    * POSSIBLE Reasons:üß†
      * BIP is too unfocused OR too broad
      * duplicated
      * disregard for formatting rules
      * technically unsound
      * NOT 
        * proper motivation OR
        * backwards compatibility OR
        * Bitcoin philosophyüß†
  * | author is ready,
    * merge the PR 
  * list it | [README.md](README.mediawiki)
  * sending email back -- to the -- BIP author / next steps
  * making strictly-editorial BIPs changes
    * _Examples:_ correcting misspellings, fixing broken links, etc.
